# ATDD Implement — 구현 + 빌드 & 테스트 (Green 단계)

이 스킬이 호출되는 시점에 **테스트 코드는 이미 존재한다.** 테스트는 컴파일이 안 되거나 실패하는 상태(Red)이며, 이 스킬의 목표는 모든 테스트를 통과(Green)시키는 프로덕션 코드를 작성하고, 빌드와 테스트로 검증하는 것이다.

**테스트 코드는 수정하지 않는다.** 테스트가 정의한 계약(인터페이스, DTO, 동작)에 맞춰 구현한다.

---

## Phase 0: 입력 검증

1. `$ARGUMENTS`에서 `{기능명}`을 파싱한다.
2. `.atdd/frd_{기능명}.md`가 존재하는지 확인한다. 없으면 에러 메시지를 출력하고 종료한다.
3. **테스트 코드가 존재하는지 확인한다.** 테스트 프로젝트에 테스트 파일이 없으면 에러를 출력하고 종료한다 — atdd-design 스킬이 먼저 실행되어야 한다.
4. `.atdd/impl_{기능명}.md`가 이미 존재하면 스킵하고 완료 신호를 출력한다.

---

## Phase 1: 분석 (코드 작성 없음)

이 Phase에서는 파일을 **읽기만** 한다. 어떤 코드 파일도 생성하거나 수정하지 않는다.

1. **테스트 코드 읽기** — 테스트 프로젝트의 테스트 파일을 모두 읽어 어떤 클래스, 메서드, DTO를 참조하는지 파악한다. 테스트가 기대하는 인터페이스가 구현의 계약이다.
2. **FRD 읽기** — `.atdd/frd_{기능명}.md`를 읽어 구현 체크리스트(시나리오, 예외, 엣지, 보안/성능 고려사항)로 사용한다.
3. **프로젝트 구조 파악** — 프레임워크, 빌드 도구, 기존 코드 패턴, 공통 모듈을 확인한다.

분석이 끝나면 아래를 출력한다:
```
[Phase 1 완료] 분석 완료 — 구현 코드 작성을 시작합니다.
```

> **⛔ Phase 1이 완료되기 전에 Phase 2로 절대 넘어가지 않는다.** 테스트 코드와 FRD를 모두 읽고, `[Phase 1 완료]`를 출력한 후에만 Phase 2를 시작한다.

---

## Phase 2: 구현 코드 작성

테스트 코드가 참조하는 모든 프로덕션 클래스를 구현하여 테스트를 통과시킨다.

### 작성 순서

1. **DTO/모델 클래스** — 테스트에서 참조하는 Request/Response/Error DTO를 먼저 생성한다.
2. **인터페이스** — 테스트에서 모킹하는 인터페이스를 생성한다.
3. **하위 레이어(Repository)부터 상위 레이어(Controller) 순서로 구현한다.**
4. **DI 등록/설정 파일** — Program.cs, Startup.cs 등 의존성 주입 설정을 마지막에 작성한다.

### 구현 규칙

- **테스트 코드를 수정하지 않는다.** 테스트가 기대하는 클래스명, 메서드 시그니처, 네임스페이스를 그대로 따른다.
- FRD에 명시되지 않은 기능을 추가하지 않는다.
- `[DEFERRED]` 항목은 TODO 주석으로 위치를 표시하고, 합리적인 기본값이나 placeholder를 사용한다.
- 보안 고려사항(FRD)을 구현에 반영한다.

---

## Phase 3: 빌드 & 테스트

구현 코드 작성이 끝나면 빌드와 테스트를 실행하여 검증한다. **최대 3회 반복한다.**

### Step 1: 빌드 실행
- 프로젝트의 빌드 명령을 실행한다 (dotnet build, npm run build, ./gradlew build 등).
- 빌드 명령은 프로젝트 설정 파일(.csproj, package.json, build.gradle 등)에서 판별한다.

→ 빌드 성공 시: Step 2로 진행한다.
→ 빌드 실패 시: 에러를 분석하고 **프로덕션 코드만** 수정한 뒤 Step 1로 돌아간다. 테스트 코드는 수정하지 않는다.

### Step 2: 테스트 실행
- 테스트 실행 명령을 실행한다 (dotnet test, npm test, ./gradlew test 등).

→ 테스트 전체 통과 시: Phase 4로 진행한다.
→ 테스트 실패 시: 실패한 테스트의 에러를 분석하고 **프로덕션 코드만** 수정한 뒤 Step 1로 돌아간다. 테스트 코드는 수정하지 않는다.

### 3회 반복 후에도 실패 시
잔여 에러 목록을 기록하고 Phase 4로 진행한다. `BUILD_TEST_FAILED` 플래그를 포함시킨다.

---

## Phase 4: 완료

구현 보고서 `.atdd/impl_{기능명}.md`를 작성하고 완료 신호를 출력한다.

### 보고서 포맷

```markdown
# 구현 완료 보고서 — {기능명}

## 생성/수정된 파일

| 파일 경로 | 작업 유형 | 설명 |
|-----------|----------|------|
| {파일 경로} | {생성/수정} | {파일의 역할과 주요 내용} |

## 빌드 & 테스트 결과

- 빌드: {성공/실패}
- 테스트 전체: {N}건
- 통과: {N}건
- 스킵(DEFERRED): {N}건
- 빌드&테스트 반복 횟수: {N}회

## DEFERRED 항목

| 항목 | 구현 영향 | 현재 처리 |
|------|----------|----------|
| {미해결 항목} | {구현에 미친 영향} | {TODO 주석 위치 또는 기본값 설명} |

## 구현 노트

{구현 중 내린 기술적 판단이 있으면 기술한다. 없으면 "없음".}
```

### 완료 신호

빌드와 테스트가 모두 성공한 경우:
```
[ATDD-IMPLEMENT COMPLETE]
- 생성 파일: {FILL: 파일 경로, 한 줄에 하나}
- 수정 파일: {FILL: 파일 경로, 한 줄에 하나. 없으면 "없음"}
- DEFERRED 처리: {N}건 (TODO 주석)
- 빌드: 성공
- 테스트: {N}건 전체 통과
```

3회 반복 후에도 빌드/테스트 실패한 경우:
```
[ATDD-IMPLEMENT COMPLETE: BUILD_TEST_FAILED]
- 생성 파일: {FILL: 파일 경로, 한 줄에 하나}
- 수정 파일: {FILL: 파일 경로, 한 줄에 하나. 없으면 "없음"}
- DEFERRED 처리: {N}건 (TODO 주석)
- 빌드: {성공/실패}
- 테스트: {통과 N}건 / {실패 N}건
- 잔여 에러:
  - {FILL: 에러 내용, 한 줄에 하나}
```

---

## Tone & Style

- 한국어를 기본 언어로 사용한다.
- 코드는 해당 기술 스택의 관례를 따른다 (네이밍, 디렉토리 구조, 패키지명 등).
- 이 스킬은 서브 에이전트에서 실행되므로 `AskUserQuestion`을 호출하지 않는다.
- 기존 코드가 있으면 그 스타일을 따른다.
