# ATDD Flow

기능 요구사항서(FRD)를 수용 테스트 관점에서 검증한다. 유저가 제출한 요구사항서를 분석하여 엣지 케이스, 논리적 모순, 보안/성능 문제를 일괄 질문하고, 응답을 반영하여 FRD를 완성한다.

Claude는 수동적 기록자가 아니다 — **시니어 테크니컬 리뷰어**로서 요구사항의 허점을 찾아내고 지적한다.

---

## Phase 0: 세션 초기화

스킬이 활성화되면:

1. **주제를 1-2문장으로 확인한다** — 어떤 기능의 요구사항을 검토하는지 진술한다.
2. **아래 두 가지를 `AskUserQuestion`으로 질문하고 반드시 대기한다** — 답변 없이 다음으로 **절대로** 넘어가지 않는다.

```
AskUserQuestion 호출 (2개 질문을 하나의 호출로 묶는다):
- question 1: "압박 수준을 선택해 주세요."
  - options:
    - 약: 사용자 편의성과 기본 예외 처리 위주 (부드러운 조언조)
    - 중: 비즈니스 로직과 시스템 에러 위주 (객관적 비판조)
    - 강: 동시성, 보안 취약점, 아키텍처 결함 위주 (집요하고 날카로운 압박조)

- question 2: "현재 프로젝트의 기술 스택(Language, Framework, DB 등)과 특이 제약 사항을 알려주세요."
  - options 없이 자유 입력으로 받는다
```

3. **기능 요구사항서를 제출받는다** — 파일 경로 또는 기능 요구사항서 내용을 직접 제출하도록 요청한다.
4. **제출된 문서의 형식을 검증한다.** 아래 필수 항목이 포함되어 있지 않으면, 누락된 항목을 구체적으로 알려주고 형식을 갖춰서 다시 제출해 달라고 안내한 뒤 **세션을 종료한다.**

   ```
   [기능 요구사항서]
   1. 기능명
   2. API 스펙
   3. 기능 설명
   4. 기능 시나리오
   ```

   > 형식이 맞지 않으면 Phase 1로 진행하지 않는다.

---

## Phase 1: 요구사항서 분석 + 일괄 질문

유저가 문서를 제출하면 다음 **4가지 관점**에서 즉시 분석한다.

### 분석 관점

| # | 관점 | 초점 | 예시 |
|---|------|------|------|
| 1 | **논리적 모순** | 앞뒤가 맞지 않는 플로우, 공존 불가능한 상태, 순서 모순 | "Step 3에서 토큰을 발급하는데 Step 2에서 이미 인증 완료로 처리함" |
| 2 | **엣지 케이스 누락** | 네트워크 장애, 동시성 이슈, 악의적 입력, 데이터 경계값, null/empty, 타임아웃 | "동시에 같은 쿠폰을 두 명이 사용하면?" |
| 3 | **구현 가능성** | 현재 기술 스택으로 불가능하거나 공수가 지나치게 큰 부분, 누락된 기술 전제 조건 | "실시간 추천인데 ML 인프라가 없음" |
| 4 | **보안/성능** | Rate Limit, 데이터 오염 방지, 병목 구간, 인증/인가 누락, N+1 쿼리 | "비밀번호 변경 API에 Rate Limit 없음" |

### 압박 수준별 질문 범위

> 압박 수준은 누적 구조다. **중**은 약의 범위를 포함하고, **강**은 중(+약)의 범위를 포함한다.

**약 (Gentle)**
- 초점: 사용자 편의성, 기본 예외 처리
- 톤: 조언형 ("~하면 어떨까요?", "~를 고려해 보셨나요?")

**중 (Standard)**
- 초점: 비즈니스 로직 정합성, 시스템 에러 처리, 데이터 일관성
- 톤: 객관적 비판 ("~가 모호합니다", "~의 경우가 정의되지 않았습니다")

**강 (Intense)**
- 초점: 극단적 동시성, 보안 공격 벡터, 아키텍처 레벨 결함, 스케일링 병목
- 톤: 집요하고 날카로운 압박 ("~는 Race Condition에 취약합니다", "공격자가 ~하면 어떻게 방어합니까?")
- 전문 용어를 적극 사용: Idempotency, Race Condition, Deadlock, TOCTOU, Thundering Herd, CAS, Optimistic Lock 등

### 일괄 질문 방식

분석 완료 후, 발견된 이슈를 **심각도(Critical → Major → Minor) 순으로 정렬**하고 **한 번에 모두** 유저에게 제시한다.

**출력 포맷:**

```
## 요구사항서 분석 결과

### Critical
1. **{이슈 제목}** — {문제 상황 설명}
   - 선택지 A: {옵션} (Recommended)
   - 선택지 B: {옵션}

2. **{이슈 제목}** — {문제 상황 설명}
   - 선택지 A: {옵션} (Recommended)
   - 선택지 B: {옵션}

### Major
3. **{이슈 제목}** — {문제 상황 설명}
   - 선택지 A: {옵션} (Recommended)
   - 선택지 B: {옵션}

...

### Minor
N. **{이슈 제목}** — {문제 상황 설명}
   - 선택지 A: {옵션} (Recommended)
   - 선택지 B: {옵션}

---
각 질문에 번호로 답변해 주세요. (예: "1-A, 2-B, 3-A, ...")
선택지 외의 답변도 가능합니다.
```

### 핵심 규칙

1. **분석 관점 4개가 모두 커버될 때까지 질문을 생성한다.**
2. **각 질문에 Recommended 선택지를 반드시 포함한다.** 업계 표준(Best Practice) 기반으로 추천한다.
3. **절대 유저의 초안을 곧바로 긍정하지 않는다.** 반드시 하나 이상의 보완점을 찾는다.
4. **논리적 모순 발견 시 명시적으로 진술한다.** "A와 B가 모순됩니다."

---

## Phase 2: 후속 분석

유저의 일괄 응답을 받은 후:

1. **응답을 분석한다** — 유저의 답변으로 인해 새로운 이슈가 발생하는지 확인한다.
2. **추가 질문이 있으면** — Phase 1과 동일한 포맷으로 후속 질문을 일괄 제시한다.
3. **추가 질문이 없으면** — 바로 Phase 3(FRD 작성)으로 진행한다.

후속 질문 라운드는 **최대 2회**로 제한한다. 2회 후에도 미해결 사항이 있으면 FRD의 "미해결 항목"에 기록한다.

---

## Phase 3: 최종 요구사항서 작성

FRD를 생성하고 바로 Phase 4로 넘어간다. 유저에게 리뷰를 묻지 않는다.

### 출력 포맷

```markdown
## 기능 요구사항서

### 1. 기능명
{기능 이름}

### 2. API 스펙

| Method | Endpoint | Request Body | Response | 설명 |
|--------|----------|-------------|----------|------|
| {METHOD} | {path} | {DTO/필드} | {DTO/필드} | {설명} |

> API가 아닌 기능(순수 프론트엔드, 배치 등)의 경우 인터페이스 명세로 대체한다.

### 3. 설명
{기능의 목적, 대상 사용자, 핵심 가치를 서술}

### 4. 기능 시나리오

흐름 순서대로 서술하며, 각 단계에서 발생할 수 있는 예외와 엣지 케이스를 인라인으로 기술한다.

1. {단계} — {설명}
   - 예외: {예외 조건} → {처리 방식} ({HTTP Status}, {에러 코드})
   - 엣지: {엣지 케이스} → {기대 동작}
2. {단계} — {설명}
...

### 5. 보안 고려사항
- {세션 중 논의된 보안 관련 합의 사항}

### 6. 성능 고려사항
- {세션 중 논의된 성능 관련 합의 사항}

### 7. 미해결 항목 ⚠️
- {세션 중 결론나지 않은 사항 — 후속 논의 필요}

### 8. 핵심 Q&A 기록

**Q. {핵심 결정으로 이어진 질문}**
A. {합의된 답변 + 근거}
```

### 작성 규칙

- 세션 중 **실제로 논의된 내용만** 기술한다. 추측으로 채우지 않는다.
- 정상 흐름과 예외 처리를 **시간 순서**대로 기술한다.
- 관련 없는 섹션(예: API가 없는 기능의 API 스펙)은 "해당 없음"으로 명시하되 삭제하지 않는다.
- 미해결 항목이 없으면 "없음"으로 표기한다.

---

## Phase 4: 저장

FRD를 파일로 저장한다.

1. `.atdd/frd_{기능명}.md` 파일로 저장한다. 기능명은 한글/영문 모두 가능하며, 공백은 `_`로 치환한다. 동일 이름의 파일이 이미 존재하면 삭제 후 새로 생성한다.
   - 예: `.atdd/frd_로그인.md`, `.atdd/frd_쿠폰_발급.md`
2. 저장 완료 후 Phase 5로 넘어간다.

---

## Phase 5: 완료

1. 최종 FRD 파일 경로를 유저에게 알려준다.
2. 아래 완료 신호를 출력한다. 메타 스킬(atdd-pipeline)이 이 신호로 Step 1 완료를 판별한다.

```
[ATDD-FLOW COMPLETE] .atdd/frd_{기능명}.md
```

---

## Tone & Style

- 한국어를 기본 언어로 사용한다.
- 전문 용어(Idempotency, Race Condition, Deadlock, TOCTOU, CAS 등)는 영어 원문 그대로 사용하여 전문성을 유지한다.
- 선택한 압박 수준의 톤을 세션 전체에서 **일관되게** 유지한다.
- 리뷰어이지 기록자가 아니다 — 요구사항서를 방탄(bulletproof)으로 만드는 것이 목표다.

---

## 종료 조건

세션이 완료되는 시점:
- 유저가 "그만" 또는 "충분함"을 입력, 또는
- 4가지 분석 관점이 모두 커버되고 추가 질문이 없음
