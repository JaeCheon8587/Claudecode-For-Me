# ATDD Design — 테스트 코드 작성 + 자기 검증 (Red 단계)

**설계 결과를 입력으로 받아** 테스트 코드를 작성하고, Claims↔테스트 교차 검증을 수행한다. 설계(Claims 추출, 레이어 구조, 테스트 전략)는 이 스킬의 범위 밖이다 — 파이프라인의 Phase 2a(Opus 메인 에이전트)가 수행하고, 그 결과를 이 스킬에 전달한다.

이 스킬의 산출물은 컴파일이 안 되는 것이 정상이다. 테스트에서 참조하는 DTO, Service, Controller 등이 아직 존재하지 않기 때문이다. 이것이 ATDD의 Red 단계다.

**프로덕션 코드는 절대 작성하지 않는다.**

---

## Phase 0: 입력 검증

1. `$ARGUMENTS`에서 FRD 파일 경로와 **설계 결과**를 파싱한다.
2. FRD 파일을 읽고 필수 섹션(기능명, API 스펙, 기능 시나리오)을 확인한다. 누락 시 에러 메시지를 출력하고 종료한다.
3. 파일명에서 `{기능명}`을 추출한다 (예: `frd_로그인.md` → `로그인`).
4. **설계 결과를 확인한다** — Claims 목록, 레이어 구조, DTO/인터페이스 시그니처, 테스트 전략이 포함되어 있어야 한다. 누락 시 에러를 출력하고 종료한다.

---

## Phase 1: 코드 탐색 (제한적)

프로젝트의 기술 스택과 컨벤션을 파악한다. **최소한의 파일만 읽는다.**

읽을 수 있는 파일:
- 프로젝트 설정 파일 **1개** (`.csproj`, `package.json`, `build.gradle` 등) — 프레임워크, 의존성 파악
- 테스트 프로젝트 설정 파일 **1개** — 테스트 프레임워크 파악
- 기존 유사 기능 파일 **최대 2개** — 네이밍 컨벤션, 레이어 패턴 파악

**⛔ 금지**: 프로젝트 전체 탐색, 디렉토리 구조 재귀 탐색, 관련 없는 파일 읽기. 위 범위를 초과하는 탐색은 하지 않는다.

```
[Phase 1 완료] 코드 탐색 완료 — 테스트 코드 작성을 시작합니다.
```

---

## Phase 2: 테스트 코드 작성

**설계 결과에서 제공된 Claims, 레이어 구조, DTO/인터페이스 시그니처, 테스트 전략을 충실히 구현한다.** 설계를 자체적으로 재해석하거나 변경하지 않는다.

### 작성 순서

1. **정상 흐름(Happy Path) 테스트부터 작성한다.**
2. 예외/엣지 케이스 테스트를 이어서 작성한다.
3. `[DEFERRED]` 테스트는 주석으로 남기고 skip 처리한다 — 삭제하지 않는다.

### 작성 규칙

- 테스트 프레임워크는 Phase 1에서 파악한 기술 스택에 맞게 선택한다 (xUnit for ASP.NET, Jest for NestJS, JUnit for Spring Boot 등).
- 테스트 파일은 **테스트 프로젝트에만** 생성한다. 프로덕션 프로젝트에 파일을 생성하지 않는다.
- 설계 결과의 테스트 전략에서 지정한 **파일 분할**을 따른다.
- 설계 결과의 **DTO/인터페이스 시그니처**를 테스트에서 그대로 참조한다 — 네임스페이스, 클래스명, 메서드 시그니처를 임의로 변경하지 않는다.
- Given/When/Then 구조를 코드 주석이나 describe/it 블록 이름으로 반영한다.
- 외부 의존성(DB, 외부 API)은 설계 결과의 인터페이스를 기반으로 모킹한다.
- FRD의 모든 시나리오 단계가 하나 이상의 테스트 케이스로 커버되어야 한다.

---

## Phase 3: 자기 검증 (Claims ↔ 테스트 교차 검증)

Phase 2에서 테스트 코드를 작성한 직후, **같은 에이전트 내에서** 설계 결과의 Claims와 테스트 코드를 매칭한다.

### Step A: Claims 확인

설계 결과에서 제공된 Claims 목록을 그대로 사용한다. Claims를 새로 추출하지 않는다.

### Step B: 테스트 매칭 (Claim ↔ 테스트 코드)

각 Claim에 대해, 방금 작성한 테스트 코드에서 해당 Claim을 검증하는 테스트 메서드를 찾는다.

매칭 기준:
- 테스트 메서드가 해당 Claim의 조건을 설정(Given)하고, 행동을 실행(When)하고, 결과를 단언(Then/Assert)하는가?
- "상태" 유형: Assert가 내부 상태(저장소 조회, 카운트 등)를 검증하는가? 응답 코드만 확인하는 것은 상태 검증이 아니다.
- "행동" 유형: 비기능적 속성(시간, 순서, 독립성)을 측정하거나 비교하는 Assert가 있는가?

---

## Phase 4: 자기 수정 루프

Phase 3에서 누락 항목이 발견되면, **테스트 코드를 직접 보완**한 뒤 Phase 3을 다시 수행한다. 기존 테스트는 유지하고, 누락된 Claim에 대한 테스트만 추가한다.

```
[반복 시작]

→ 누락 없음: 루프 종료, Phase 5(완료)으로 진행한다.
→ 누락 있음: 누락된 Claim에 대한 테스트를 추가하고, Phase 3을 다시 수행한다.

[반복 끝 — 누락 없음일 때까지, 최대 3회]
```

### 무한 루프 방지

Phase 3↔4 반복이 **3회를 초과하면**, 잔여 누락 목록을 포함하여 완료한다. `GAPS_REMAIN` 플래그를 출력에 포함시킨다.

---

## Phase 5: 완료

테스트 파일 작성과 자기 검증이 끝나면 아래 스켈레톤의 `{FILL}` 부분만 채워서 출력한다. 구조를 변경하거나 추가 산문을 작성하지 않는다.

자기 검증을 통과한 경우:
```
[ATDD-DESIGN COMPLETE]
- 파일: {FILL: 테스트 파일 경로, 한 줄에 하나}
- 테스트: {N}건 (DEFERRED 스킵: {N}건)
- 커버리지: Happy Path {N}건, 예외 {N}건, 엣지 {N}건
- 자기 검증: 통과
- 검증 반복: {N}회
```

잔여 누락이 남은 경우 (3회 초과):
```
[ATDD-DESIGN COMPLETE: GAPS_REMAIN]
- 파일: {FILL: 테스트 파일 경로, 한 줄에 하나}
- 테스트: {N}건 (DEFERRED 스킵: {N}건)
- 커버리지: Happy Path {N}건, 예외 {N}건, 엣지 {N}건
- 자기 검증: 잔여 누락 {N}건
- 검증 반복: {N}회
- 누락 목록:
  - {FILL: 누락된 Claim, 한 줄에 하나}
```

## Tone & Style

- 한국어를 기본 언어로 사용한다.
- 코드는 해당 기술 스택의 관례를 따른다.
- 이 스킬은 서브 에이전트에서 실행되므로 `AskUserQuestion`을 호출하지 않는다.
- 기존 테스트 코드가 있으면 그 스타일을 따른다.
- **설계 결과를 신뢰한다** — Claims, 시그니처, 테스트 전략을 임의로 변경하지 않는다.
