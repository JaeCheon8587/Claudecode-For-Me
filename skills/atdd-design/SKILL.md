# ATDD Design — 인터페이스 설계 + 수용 테스트 명세

FRD(기능 요구사항서)를 입력받아 두 개의 산출물을 생성한다:
1. **인터페이스 설계서** — 레이어 구조, DTO, 메서드 시그니처, 의존성 맵
2. **수용 테스트 명세** — Given/When/Then 테스트 케이스 + FRD 커버리지 매핑

설계 관점은 **Software Architect** — 구현 세부사항이 아닌 인터페이스 계약과 레이어 경계에 집중한다. 구현 코드는 작성하지 않는다.

---

## Phase 0: 입력 검증

1. `$ARGUMENTS`에서 FRD 파일 경로를 파싱한다.
2. FRD 파일을 읽고 필수 섹션이 있는지 확인한다: 기능명, API 스펙, 기능 시나리오.
   - 누락 시 에러 메시지를 출력하고 종료한다.
3. FRD 파일명에서 `{기능명}`을 추출한다 (예: `frd_로그인.md` → `로그인`).
4. `.atdd/` 폴더에 이미 `design_{기능명}.md`와 `test_{기능명}.md`가 존재하면 스킵하고 완료 신호를 출력한다.

---

## Phase 1: FRD 분석

FRD를 읽고 아래 항목을 내부적으로 정리한다 (유저에게 출력하지 않음):

- **도메인 모델**: 핵심 엔티티, 값 객체, 관계
- **명령(Command)과 쿼리(Query)**: 시나리오에서 추출한 시스템 동작
- **불변식(Invariant)**: 시나리오와 예외에서 추출한 비즈니스 규칙
- **미해결 항목**: FRD의 "미해결 항목" 섹션 → `[DEFERRED]`로 표기할 대상

기술 스택 정보가 FRD에 포함되어 있으면 (예: NestJS, Spring Boot) 해당 프레임워크의 관례에 맞게 레이어와 네이밍을 조정한다.

---

## Phase 2: 인터페이스 설계서 작성

아래 포맷으로 `.atdd/design_{기능명}.md`를 작성한다.

```markdown
# 인터페이스 설계서 — {기능명}

## 입력
- FRD: .atdd/frd_{기능명}.md

## 1. 레이어 구조

각 레이어의 이름, 책임, 포함할 클래스/모듈을 나열한다.

| 레이어 | 책임 | 주요 클래스/모듈 |
|--------|------|-----------------|
| {Controller/Presentation} | {HTTP 요청 수신, 유효성 검사, 응답 변환} | {클래스명} |
| {Service/Application} | {비즈니스 로직, 트랜잭션 경계} | {클래스명} |
| {Repository/Infrastructure} | {데이터 접근, 외부 시스템 연동} | {클래스명} |

## 2. DTO 정의

### Request DTO
{필드명, 타입, 유효성 규칙, 필수 여부}

### Response DTO
{필드명, 타입, 설명}

### Error Response DTO
{에러 코드, HTTP 상태, 메시지 패턴}

## 3. 메서드 시그니처

각 레이어의 public 메서드를 나열한다. 구현 본문은 작성하지 않는다.

### {레이어명}
- `{반환타입} {메서드명}({파라미터}): {설명}`

## 4. 의존성 맵

클래스 간 의존 방향을 나열한다. 상위 레이어 → 하위 레이어 방향만 허용한다.

{ClassA} → {ClassB}: {의존 이유}

## DEFERRED 항목
- {FRD 미해결 항목 중 설계에 영향을 줄 수 있는 것} [DEFERRED]
```

### 설계 규칙

- FRD의 API 스펙과 시나리오에 기반하여 작성한다. FRD에 없는 기능을 추가하지 않는다.
- 레이어 간 의존 방향은 단방향(상위→하위)만 허용한다.
- DTO는 FRD의 Request/Response 스펙에서 직접 도출한다.
- 미해결 항목은 임의로 결정하지 않고 `[DEFERRED]`로 표기한다.

## Phase 3: 수용 테스트 명세 작성

설계서 작성 후, 아래 포맷으로 `.atdd/test_{기능명}.md`를 작성한다.

```markdown
# 수용 테스트 명세 — {기능명}

## 입력
- FRD: .atdd/frd_{기능명}.md
- 설계서: .atdd/design_{기능명}.md

## 테스트 케이스

### TC-01: {시나리오 이름}
- **Given**: {전제 조건 — 초기 상태, 데이터 셋업}
- **When**: {행위 — API 호출 또는 시스템 동작}
- **Then**: {기대 결과 — 응답, 상태 변화, 부수 효과}

### TC-02: {시나리오 이름}
- **Given**: {전제 조건}
- **When**: {행위}
- **Then**: {기대 결과}

...

## 커버리지 매핑

| FRD 시나리오 단계 | 테스트 케이스 |
|-------------------|--------------|
| {단계 1 — 설명} | TC-01 |
| {단계 1 예외 — 빈값 입력} | TC-02 |
| {단계 2 — 설명} | TC-03 |
| ... | ... |

## DEFERRED 항목
- {FRD 미해결 항목 중 테스트에 영향을 주는 것} [DEFERRED]
```

### 테스트 작성 규칙

- FRD의 시나리오 단계, 예외, 엣지 케이스를 **빠짐없이** 테스트 케이스로 변환한다.
- 정상 흐름(Happy Path)을 먼저 작성하고, 예외/엣지 케이스를 이어서 작성한다.
- Given/When/Then은 구체적으로 작성한다 — "유효한 데이터"가 아닌 실제 값의 형태를 서술한다.
- 커버리지 매핑에서 FRD의 모든 시나리오 단계가 하나 이상의 TC와 연결되어야 한다. 누락이 있으면 TC를 추가한다.
- FRD의 미해결 항목에 해당하는 테스트는 `[DEFERRED]`로 표기하고, 전제 조건을 가정하여 작성하되 가정 사항을 명시한다.

---

## Phase 4: 저장 및 완료

1. 두 파일을 저장한다:
   - `.atdd/design_{기능명}.md`
   - `.atdd/test_{기능명}.md`
2. 완료 신호를 출력한다:

```
[ATDD-DESIGN COMPLETE] .atdd/design_{기능명}.md, .atdd/test_{기능명}.md
```

---

## Tone & Style

- 한국어를 기본 언어로 사용한다.
- 코드/기술 용어는 영어 원문을 사용한다 (DTO, Controller, Repository 등).
- 설계서는 간결하고 구조적으로 작성한다 — 산문체가 아닌 표/목록 중심.
- 이 스킬은 서브 에이전트에서 실행될 수 있으므로 `AskUserQuestion`을 호출하지 않는다. 판단이 필요한 사항은 FRD에 근거하여 결정하거나 `[DEFERRED]`로 남긴다.
