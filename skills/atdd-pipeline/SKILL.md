# ATDD Pipeline — 오케스트레이터

ATDD 개발 파이프라인의 전체 흐름을 관리한다. 직접 설계나 구현을 수행하지 않고, 각 Phase 스킬에 위임한다.

## 핵심 원칙

- **파일 기반 계약**: Phase 간 데이터 전달은 `.atdd/` 폴더의 파일과 소스 코드로만 이루어진다. 컨텍스트 변수나 메모리에 의존하지 않는다.
- **재개 가능**: 산출물 존재 여부로 완료를 판별한다. 이미 완료된 Phase는 스킵한다.
- **Phase 1은 메인 에이전트(Opus)**: 유저와 대화가 필요하므로 Opus가 직접 수행한다.
- **Phase 2, 3은 서브 에이전트(Sonnet)**: FRD를 읽고 코드를 작성하는 구현이므로 Sonnet에 위임한다. Phase 2(테스트 코드+자기검증)와 Phase 3(구현+빌드&테스트)은 별도 서브 에이전트로 분리하여 Red→Green 순서를 구조적으로 강제한다.
- **토큰 효율**: 서브 에이전트에 파일 내용을 프롬프트로 복사하지 않는다. 파일 경로만 전달하고 서브 에이전트가 직접 읽게 한다. 대화에는 요약만 출력한다.

## 산출물 컨벤션

| Phase | 파일명 패턴 | 설명 |
|-------|------------|------|
| Phase 1 (Q&A 임시) | `frd_{기능명}_temp.md` | 진행 중 실시간 Q&A 기록 |
| Phase 1 (FRD) | `frd_{기능명}.md` | 완성된 기능 요구사항서 |
| Phase 2 | 테스트 코드 파일 | 테스트 프로젝트에 생성 |
| Phase 3 (보고서) | `impl_{기능명}.md` | 구현 완료 보고서 (빌드/테스트 결과 포함) |

`{기능명}`은 Phase 1에서 결정되며, 모든 Phase에서 동일하게 사용한다.

---

## Phase 0: 초기화

1. `$ARGUMENTS`를 파싱한다:
   - 파일 경로가 주어졌으면 → FRD 또는 요구사항서 파일 경로로 사용
   - 텍스트가 주어졌으면 → 기능 요구사항서 내용으로 사용
   - 인자가 없으면 → Phase 1에서 유저에게 요청

2. `.atdd/` 폴더를 확인하여 이미 완료된 Phase를 판별한다:

   ```
   frd_{기능명}.md 존재?       → Phase 1 완료
   테스트 코드 파일 존재?      → Phase 2 완료
   impl_{기능명}.md 존재?      → Phase 3 완료
   ```

3. **기존 산출물이 하나라도 존재하면**, `AskUserQuestion`으로 유저에게 묻는다:

   ```
   AskUserQuestion:
   - question: "기존 산출물이 발견되었습니다. 어떻게 진행할까요?"
   - options:
     - "이어서 진행 (완료된 Phase 스킵)" (Recommended)
     - "처음부터 시작 (기존 산출물 전부 삭제)"
   ```

   - **이어서 진행** → 기존 산출물을 유지하고, 미완료 Phase부터 재개한다.
   - **처음부터 시작** → `.atdd/` 폴더 내 해당 기능명의 산출물(`frd_`, `impl_`, `frd_*_temp`)을 모두 삭제한 뒤 Phase 1부터 시작한다.

4. 이미 FRD 파일이 존재하고 이어서 진행하는 경우, 기능명을 파일명에서 추출한다.
5. 현재 상태를 유저에게 보고한다:

   ```
   [ATDD Pipeline] 상태 확인
   - Phase 1 (FRD): {완료 ✓ / 미완료}
   - Phase 2 (테스트 코드+자기검증): {완료 ✓ / 미완료}
   - Phase 3 (구현+빌드&테스트): {완료 ✓ / 미완료}
   → Phase {N}부터 시작합니다.
   ```

---

## Phase 1: FRD 생성 (메인 에이전트)

### 완료 조건
`.atdd/frd_{기능명}.md` 존재

### 이미 완료된 경우
스킵하고 Phase 2로 진행한다.

### 미완료인 경우

`skills/atdd-flow/SKILL.md`의 지침을 **메인 에이전트에서 직접** 수행한다.

Phase 1은 유저와의 소크라틱 대화가 핵심이므로 서브 에이전트에 위임하지 않는다. atdd-flow 스킬의 Phase 0~5를 그대로 따른다.

FRD가 생성되면 (`[ATDD-FLOW COMPLETE]` 신호 확인) **유저 컨펌 게이트**로 진입한다. Phase 1이 완료되기 전까지 **절대로** Phase 2로 넘어가지 않는다.

### 유저 컨펌 게이트 (Phase 1 → Phase 2)

FRD가 생성된 후, 유저에게 확인을 요청한다. 유저가 "Phase 2로 진행"을 선택할 때까지 이 루프를 반복한다.

```
[반복 시작]

AskUserQuestion:
- question: "FRD가 완성되었습니다. 검토 후 다음 단계로 진행할까요?"
- options:
  - "Phase 2로 진행 (테스트 코드 작성)" (Recommended)
  - "피드백 있음 — FRD 수정 필요"

→ "Phase 2로 진행" 선택 시: 루프 종료, Phase 2로 진행한다.
→ "피드백 있음" 선택 시: 유저의 피드백을 받아 FRD를 수정하고, 수정 완료 후 다시 AskUserQuestion으로 컨펌을 요청한다.

[반복 끝 — 유저가 "Phase 2로 진행"을 선택할 때까지]
```

---

## Phase 2: 테스트 코드 작성 + 자기 검증 (서브 에이전트 — Sonnet)

### 선행 조건
`.atdd/frd_{기능명}.md` 존재

### 이미 완료된 경우
테스트 코드가 이미 존재하면 Phase 2를 스킵하고 Phase 3으로 진행한다.

### 미완료인 경우

**Agent 도구**를 사용하여 서브 에이전트를 호출한다. Agent 도구의 **model 파라미터를 "sonnet"**으로 지정한다. 서브 에이전트에게 아래 지시를 전달한다:

```
skills/atdd-design/SKILL.md를 읽고, 이 스킬의 지침을 수행하라.

인자: .atdd/frd_{기능명}.md
```

서브 에이전트는 **FRD를 직접 읽어** 테스트 코드를 작성하고, FRD 시나리오↔테스트 교차 검증(자기 검증)을 수행한다. 프로덕션 코드는 작성하지 않는다 — Red 단계다.

> ⛔ 서브 에이전트 프롬프트에 FRD 내용을 복사하지 않는다. 파일 경로만 전달한다.

서브 에이전트의 완료를 기다린다. 완료 신호를 확인한다:

- `[ATDD-DESIGN COMPLETE]` → 자기 검증 통과. Phase 3으로 **자동 진행**한다.
- `[ATDD-DESIGN COMPLETE: GAPS_REMAIN]` → 자기 검증 3회 초과. 잔여 누락 목록을 유저에게 보고하고 판단을 요청한다:

```
AskUserQuestion:
- question: "테스트 자기 검증이 3회 반복 후에도 {N}건의 누락이 남아 있습니다. 어떻게 진행할까요?"
- options:
  - "현재 상태로 Phase 3 진행" (Recommended)
  - "Phase 2 서브 에이전트 재호출 (누락 항목 포함)"
  - "중단"
```

Phase 2 완료 후 **Phase 3으로 자동 진행한다** — 유저 컨펌 게이트 없이 바로 넘어간다. 자기 검증을 통과했으므로 테스트 코드의 품질은 이미 보장되었다.

---

## Phase 3: 구현 + 빌드 & 테스트 (서브 에이전트 — Sonnet)

### 선행 조건
Phase 2에서 테스트 코드가 생성되어 있어야 한다.

### 완료 조건
`.atdd/impl_{기능명}.md` 존재

### 이미 완료된 경우
스킵하고 Phase 4(완료)로 진행한다.

### 미완료인 경우

**Agent 도구**를 사용하여 서브 에이전트를 호출한다. Agent 도구의 **model 파라미터를 "sonnet"**으로 지정한다. 서브 에이전트에게 아래 지시를 전달한다:

```
skills/atdd-implement/SKILL.md를 읽고, 이 스킬의 지침을 수행하라.

인자: {기능명}
FRD 파일: .atdd/frd_{기능명}.md
```

이 서브 에이전트는 **테스트를 통과시키는 프로덕션 코드를 작성하고, 빌드와 테스트를 실행하여 검증한다.** 테스트 코드는 이미 존재하므로 수정하지 않는다.

> ⛔ 서브 에이전트 프롬프트에 FRD 내용이나 테스트 코드를 복사하지 않는다. 파일 경로만 전달한다.

서브 에이전트의 완료를 기다린다. 완료 신호를 확인한다:

- `[ATDD-IMPLEMENT COMPLETE]` → 구현 + 빌드 + 테스트 모두 성공. Phase 4로 진행한다.
- `[ATDD-IMPLEMENT COMPLETE: BUILD_TEST_FAILED]` → 3회 반복 후에도 빌드/테스트 실패. 실패 내역을 유저에게 보고한다.

---

## Phase 4: 완료

모든 Phase가 완료되면 산출물을 보고한다.

`.atdd/impl_{기능명}.md`를 읽어 빌드/테스트 결과를 확인한다 (이 보고서는 Phase 3의 서브 에이전트가 생성한 것이다).

최종 산출물 보고:

```
[ATDD Pipeline 완료]

산출물:
1. FRD:        .atdd/frd_{기능명}.md
2. 테스트 코드: {Phase 2에서 생성된 테스트 파일 목록}
3. 구현 코드:  {Phase 3에서 생성된 파일 목록}
4. 구현 보고서: .atdd/impl_{기능명}.md
```

> ⛔ impl 보고서를 여기서 새로 작성하지 않는다 — Phase 3 서브 에이전트가 이미 생성한 것을 그대로 사용한다.

---

## 에이전트 페르소나 배정

| Phase | 스킬 | 실행 위치 | 모델 | 근거 |
|-------|------|----------|------|------|
| Phase 1 (FRD) | atdd-flow | 메인 에이전트 | Opus | 유저와 직접 대화하며 요구사항 정제 |
| Phase 2 (테스트 코드+자기검증) | atdd-design | 서브 에이전트 | Sonnet | FRD를 읽어 테스트 코드 작성 + 커버리지 검증 |
| Phase 3 (구현+빌드&테스트) | atdd-implement | 서브 에이전트 | Sonnet | FRD + 테스트 코드를 참조하여 구현 후 빌드/테스트 검증 |

---

## 에러 처리

- 서브 에이전트가 실패하면 에러 내용을 유저에게 보고하고, 해당 Phase를 재시도할지 묻는다.
- 선행 조건 파일이 없으면 이전 Phase부터 실행해야 함을 안내한다.
- `.atdd/` 폴더가 없으면 자동 생성한다.

---

## Tone & Style

- 한국어를 기본 언어로 사용한다.
- 오케스트레이터는 간결하게 상태를 보고한다 — 각 Phase의 세부 내용은 해당 스킬이 처리한다.
- Phase 전환 시 현재 진행 상황을 한 줄로 안내한다.
- 서브 에이전트 완료 후 테스트 파일이나 구현 파일의 전문을 메인 컨텍스트로 읽어오지 않는다 — 서브 에이전트의 완료 보고만 확인한다.
